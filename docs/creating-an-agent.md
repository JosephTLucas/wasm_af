# Creating a New Agent

This guide covers two paths for adding an agent to WASM_AF:

- **Platform agent** (sections 1-6) — a Rust crate in the `components/` workspace, compiled to a WASM component at build time, with full access to WIT-typed host interfaces and a tailored OPA policy.
- **External agent (BYOA)** (section 7) — a pre-compiled `.wasm` component uploaded to a running orchestrator via `POST /agents`, automatically sandboxed with the restrictive "untrusted" policy tier.

## Overview

Every agent in WASM_AF is a WebAssembly Component that exports a single `execute` function. The orchestrator creates a fresh wasmtime instance for each invocation, links only the host interfaces that OPA permits, calls `execute`, reads the result, and drops the instance.

```
TaskInput (WIT typed) → execute() → TaskOutput (WIT typed)
```

The agent contract is defined in `wit/agent.wit`. Agents implement the `Guest` trait generated by `wit-bindgen` and use WIT-typed imports for host functions.

## 1. Create the Rust crate

From the `components/` directory:

```bash
cargo init --lib agents/my-agent
```

Add the crate to the workspace in `components/Cargo.toml`:

```toml
[workspace]
members = [
    "agents/my-agent",
    # ...existing members...
]
```

Set up `agents/my-agent/Cargo.toml`:

```toml
[package]
name = "my-agent"
version = "0.1.0"
edition = "2021"

[lib]
crate-type = ["cdylib"]

[dependencies]
wit-bindgen = "0.39"
serde = { version = "1", features = ["derive"] }
serde_json = "1"

[package.metadata.component]
package = "wasm-af:agent"
```

The default target (`wasm32-wasip2`) is set in `components/.cargo/config.toml`.

## 2. Implement the execute function

Create `agents/my-agent/src/lib.rs`:

```rust
wit_bindgen::generate!({
    world: "agent",
    path: "../../../wit/agent.wit",
});

struct MyAgent;

export!(MyAgent);

#[derive(serde::Deserialize)]
struct MyRequest {
    query: String,
}

#[derive(serde::Serialize)]
struct MyResponse {
    result: String,
}

impl Guest for MyAgent {
    fn execute(input: TaskInput) -> Result<TaskOutput, String> {
        let req: MyRequest = serde_json::from_str(&input.payload)
            .map_err(|e| format!("payload parse error: {e}"))?;

        if req.query.trim().is_empty() {
            return Err("query is required".to_string());
        }

        let response = MyResponse {
            result: format!("processed: {}", req.query),
        };

        let payload = serde_json::to_string(&response)
            .map_err(|e| format!("serialization error: {e}"))?;

        Ok(TaskOutput {
            payload,
            metadata: vec![],
        })
    }
}
```

### Using host functions

Host functions are WIT-typed imports generated by `wit-bindgen`. Call them directly — no `unsafe`, no JSON serialization at the boundary:

```rust
use wasm_af::agent::host_llm;

let resp = host_llm::llm_complete(&host_llm::LlmRequest {
    model: String::new(),
    messages: vec![host_llm::LlmMessage {
        role: "user".to_string(),
        content: "Hello".to_string(),
    }],
    max_tokens: 256,
    temperature: Some(0.7),
})?;
```

Available host interfaces (see `wit/agent.wit` for full type definitions):

| Interface | Function | Import path |
|---|---|---|
| `host-llm` | `llm-complete` | `wasm_af::agent::host_llm::llm_complete` |
| `host-kv` | `kv-get`, `kv-put` | `wasm_af::agent::host_kv::{kv_get, kv_put}` |
| `host-exec` | `exec-command` | `wasm_af::agent::host_exec::exec_command` |
| `host-sandbox` | `sandbox-exec` | `wasm_af::agent::host_sandbox::sandbox_exec` |
| `host-email` | `send-email` | `wasm_af::agent::host_email::send_email` |
| `host-config` | `get-config` | `wasm_af::agent::host_config::get_config` |

### Using HTTP

For agents that need outbound HTTP (like `url-fetch` and `web-search`), use the `waki` crate which wraps `wasi:http/outgoing-handler`:

```toml
[dependencies]
waki = "0.5"
```

```rust
use waki::Client;

let resp = Client::new()
    .get("https://example.com")
    .header("Accept", "text/html")
    .send()
    .map_err(|e| format!("fetch failed: {e}"))?;
```

The orchestrator's `WasiHttpView` enforces `allowed_hosts` — requests to unlisted domains are rejected before reaching the network.

### Using WASI filesystem

Agents that need filesystem access (like `file-ops`) use standard `std::fs` operations. The orchestrator mounts allowed host paths via `WasiCtxBuilder::preopened_dir()`, controlled by OPA's `allowed_paths` policy decision.

## 3. Register the agent

Add an entry to your example's `agents.json`:

```json
{
  "my-agent": {
    "wasm_name": "my_agent",
    "capability": "custom",
    "context_key": "my_agent_result",
    "host_functions": [],
    "payload_fields": {
      "query": "task.context.query"
    }
  }
}
```

For agents that need HTTP, include `"http"` in `host_functions`:

```json
"host_functions": ["http"]
```

## 4. Write the OPA policy

Add allow rules in your example's `policy.rego`:

```rego
import rego.v1

allow if {
    input.step.agent_type == "my-agent"
    input.agent.capability == "custom"
}
```

For agents with host functions or HTTP:

```rego
host_functions := ["llm_complete"] if {
    input.step.agent_type == "my-agent"
}

allowed_hosts := ["api.example.com"] if {
    input.step.agent_type == "my-agent"
}
```

## 5. Build and test

```bash
# Build the component
cd components && cargo build --release -p my-agent

# OPA policy tests
opa test examples/your-example/ -v

# Integration test via curl
curl -X POST localhost:8080/tasks \
  -H 'Content-Type: application/json' \
  -d '{"type":"generic","query":"test","context":{"steps":"[{\"agent_type\":\"my-agent\",\"params\":{\"query\":\"hello\"}}]"}}'
```

The output will be at `components/target/wasm32-wasip2/release/my_agent.wasm`.

## 6. Key design principles

- **Validate inputs early.** Check for empty/invalid fields before calling host functions.
- **Return errors as `Err(String)`.** The orchestrator captures the error and marks the step as failed.
- **Never panic.** Use `Result` for all fallible operations.
- **Credentials stay in the host.** API keys live in host function closures or OPA config injection, never in WASM memory.

## 7. External agent (BYOA)

Any WASM component that implements the `wasm-af:agent` WIT world can be uploaded at runtime. Components can be built with any language toolchain that targets the Component Model:

- **Rust** via `cargo build --target wasm32-wasip2` with `wit-bindgen`
- **Python** via `componentize-py`
- **Go** via TinyGo + `componentize-go`
- **JavaScript** via `jco`

### Python agent walkthrough

Install `componentize-py`:

```bash
pip3 install componentize-py
```

Generate the Python bindings from the WIT definition:

```bash
mkdir my-agent && cd my-agent
componentize-py -d ../wit/agent.wit -w agent-untrusted bindings .
```

This produces a `wit_world/` package with the types and protocol your code must implement. The generated `WitWorld` protocol has one method — `execute` — that receives a `TaskInput` and returns a `TaskOutput`.

Write your agent in `app.py`:

```python
import json
from wit_world.imports.types import TaskInput, TaskOutput, KvPair

class WitWorld:
    def execute(self, input: TaskInput) -> TaskOutput:
        # Read data from ancestor steps via input.context,
        # or from payload fields via input.payload.
        text = ""
        for kv in input.context:
            text += kv.val

        # Pure computation — no host functions available.
        result = {"processed": len(text), "status": "ok"}

        return TaskOutput(
            payload=json.dumps(result),
            metadata=[KvPair(key="status", val="ok")],
        )
```

Key points:
- The class **must** be named `WitWorld` (matching the generated protocol).
- Return `TaskOutput` for success; raise any exception for `Err(string)`.
- `input.context` contains outputs from ancestor steps (keyed by their `context_key`).
- `input.payload` is a JSON string built from the agent's `payload_fields` mapping.
- Only `host-config` is available. No LLM, HTTP, shell, KV, or email imports.

Build the WASM component:

```bash
componentize-py -d ../wit/agent.wit -w agent-untrusted componentize app -o my_agent.wasm
```

> **macOS 26+**: `componentize-py` may crash due to Mach port guard restrictions in wasmtime. Use Docker as a workaround:
> ```bash
> docker run --rm -v "$PWD:/agent" -v "$PWD/../wit:/wit:ro" -w /agent \
>   python:3.12-slim bash -c \
>   'pip install componentize-py && componentize-py -d /wit -w agent-untrusted bindings . && componentize-py -d /wit -w agent-untrusted componentize app -o my_agent.wasm'
> ```

Python components are ~40 MB (they embed CPython). The orchestrator accepts uploads up to 50 MiB.

### Upload

```bash
curl -X POST localhost:8080/agents \
  -F 'meta={"name":"my-agent","context_key":"my_agent_result"}' \
  -F 'wasm=@path/to/my_agent.wasm'
```

The orchestrator validates the component against the WIT world (checks that `execute` is exported with the correct type signature), writes it to `WASM_DIR/external/`, and registers it with `capability: "untrusted"`.

### BYOA policy

External agents are governed by the BYOA policy tier (`policies/byoa.rego`). Default sandbox:

| Constraint | Default |
|---|---|
| Host functions | none |
| Network (HTTP) | none |
| Memory | 64 pages (4 MiB) — increase to 256 for Python agents |
| Timeout | 10 seconds |
| Human approval | required |

### Approve for execution

```json
{
  "config": {
    "approved_external_agents": ["my-agent"]
  }
}
```

### List and remove

```bash
curl localhost:8080/agents | jq '.[] | select(.external)'
curl -X DELETE localhost:8080/agents/my-agent
```
